Abstract

This work presents a year-long contribution to the Metasploit Framework, focusing on documentation, code review, and vulnerability exploitation. Efforts included creating official module documentation, improving password-cracking utilities, and enhancing usability through targeted fixes. Practical skills were developed by reproducing and documenting real-world vulnerabilities on Metasploitable3, including remote code execution, SQL injection, and credential exposure. The project demonstrates the technical and collaborative expertise required for effective participation in a large-scale, open-source cybersecurity tool.

1 Introduction

This project focuses on contributing to the open-source community through the Metasploit Framework, a widely used platform for penetration testing and exploit development. The goal is to collaborate by improving module documentation, following community guidelines, and gaining hands-on experience with cybersecurity tools and open-source workflows.

2 Project Description

As I said bebore, The Metasploit Framework is one of the most well-known open source penetration testing platforms. Maintained by Rapid7 and a global community of contributors, it aims to provide security professionals, researchers, and enthusiasts with a modular and extensible environment for developing, testing, and executing exploit code against remote targets.

2.1 Objectives

The main objectives of the Metasploit Framework are:

  • To offer a comprehensive suite of tools for penetration testing, vulnerability validation, and security auditing.

  • To enable the creation and sharing of modular exploit code, payloads, auxiliary modules, and post-exploitation tools.

  • To serve as an educational and training resource for learning about system vulnerabilities and exploitation techniques.

2.2 Scope

The scope of the project includes:

  • A large collection of pre-built modules for exploiting known vulnerabilities across multiple platforms (Windows, Linux, macOS, etc.).

  • A robust API that allows for the development of custom modules and automation.

  • Integration with other tools and frameworks (e.g., Nmap, Nessus, Nexpose).

  • A built-in console interface (msfconsole) and various payload delivery methods.

2.3 Challenges

Some of the key challenges of contributing to Metasploit include:

  • Ensuring code quality, compatibility, and security across a wide range of supported modules and environments.

  • Keeping up with newly discovered vulnerabilities and rapidly evolving attack techniques.

  • Following strict contribution guidelines and maintaining clear, validated documentation.

  • Understanding complex low-level networking, exploitation concepts, and Ruby programming.

Today, while browsing through open issues and pull requests to find ways to contribute, I came across an issue that caught my attention and seemed solvable. The problem was related to a password cracking module within Metasploit that failed under specific conditions. Specifically, the failure occurred when a user had only one of the required binary tools installed—either john or hashcat—but not both. The existing code expected both binaries to be present or forced the user to install the missing one, which was inconvenient and led to errors during execution.


Figure 6: Metasploit Issue #20396

The issue was that the system checked strictly for the presence of both binaries and threw an error when only one was available, displaying the message:

"No suitable john/hashcat binary was found on the system"

To address this, I implemented a feature that allows the module to automatically select the available binary instead of requiring both. This “auto” mode prioritizes using hashcat if present, and if not, it falls back to john. Only if neither binary is found does it return an error. This approach improves the module’s flexibility and usability, allowing it to work seamlessly in environments where only one of the binaries is installed.


Figure 7: Main changes to close the issue #20396

With this enhancement, users no longer need to manually install both binaries to run the password cracking functionality, which streamlines the user experience and reduces setup complexity.

Also, While reviewing another pull request, I suggested adding a clarification to ensure testers download a plugin version that is confirmed to be vulnerable at the “Vulnerable Application” section of the documentation. Specifically, I proposed the note:

"Ensure you are downloading a version that is confirmed vulnerable (<= 1.1.4)."

This addition helps prevent confusion when newer, patched versions are used, and ensures that testing scenarios reproduce the intended vulnerability. I also acknowledged that the existing instructions were already clear and well-written, and expressed appreciation for the contributor’s original work.

3.3 Learning

One of the most basic aspects of this project is learning how to use Metasploit effectively. To achieve this, I spent several weeks solely researching and watching [?] about the framework to understand its full potential. If I needed to resume it, this aspects would be the most important ones:

  • Workspaces
    Workspaces allow you to organize your data by separating different sessions or projects. They store information such as hosts, services, and credentials independently. Some useful commands include:

    • workspace – lists current workspaces.

    • workspace -a <name> – creates a new workspace.

    • workspace <name> – switches to a specific workspace.

  • Essential Metasploit Commands
    These are the commands I used most often while navigating and using Metasploit:

    • search <keyword> – finds relevant modules.

    • use <module> – loads the desired module.

    • info – displays details about the module.

    • show options – shows required/optional parameters.

    • set <option> <value> – configures module options.

    • run or exploit – executes the module.

    • sessions – lists active sessions.

    • sessions -i <id> – interacts with a session.

  • Module Configuration
    Every module needs to be properly configured before it can be run. This includes setting target hosts, ports, payloads, and network settings:

    • RHOSTS – the IP address of the target machine.

    • RPORT – the port used by the target service.

    • PAYLOAD – the code to be executed after exploitation.

    • LHOST / LPORT – local machine IP and port for reverse connections.

    • show advanced – displays additional configurable parameters.

  • Integration with Nmap
    Nmap, a popular network scanner, can be directly integrated with Metasploit to import scan results and quickly identify potential targets and vulnerabilities. Using the command db_nmap, the results are stored in the Metasploit database and can be analyzed using hosts, services, and vulns commands. This improves reconnaissance efficiency and streamlines the exploitation workflow.

  • Meterpreter
    Meterpreter is an advanced, dynamic payload used after a successful exploit. It provides an interactive shell that runs in memory, avoiding writing to disk and making detection harder. It allows the attacker to perform post-exploitation tasks such as privilege escalation, file system browsing, keystroke logging, and webcam/audio recording. It supports extensions and works over encrypted channels.

3.4 Metasploitable3

As part of the hands-on learning phase, I installed and worked with Metasploitable3, a deliberately vulnerable virtual machine maintained by Rapid7. 3.4.1 ProFTPD 1.3.5 (port 21) - Remote Code Execution via mod_copy

Through an Nmap scan, I identified which ports were open. I notice that the port 21 was one of them and running ProFTPD version 1.3.5, a known vulnerable FTP server. This version is affected by a Remote Code Execution (RCE) vulnerability via the mod_copy module, which allows unauthenticated attackers to copy files from one location on the server to another, enabling arbitrary command execution.

3.4.2 Payroll Application - SQL Injection (UNION-based)

Later in the engagement, I analyzed a web application available at /payroll_app.php. During testing, I discovered that the login functionality of this application was vulnerable to a UNION-based SQL injection in the password parameter.

This vulnerability was first confirmed manually using Burp Suite Repeater, by modifying the POST request and observing unexpected behavior in the response. Specifically, inserting the character * as a marker in the password field revealed a potentially injectable entry point.

To automate exploitation and extract data, I used sqlmap, a powerful tool for detecting and exploiting SQL injection vulnerabilities. The request was saved as req.txt and the following command was executed: 

sqlmap -r req.txt --dbms="SQL" --level=1 --risk=1 --dump --output-dir=output --batch

  • --dbms: Indicates the type of Database Management System expected on the server. In this case, the scan was tailored for MySQL, optimizing payloads and queries accordingly.

  • --level=1: Sets the depth of the tests to be performed. A level of 1 means that only the most basic and least intrusive checks are executed.

  • --risk=1: Specifies the potential impact of the payloads used. Risk level 1 ensures a cautious approach, reducing the chance of affecting the server’s stability.

  • --dump: Instructs SQLMap to retrieve and display the contents of the discovered databases.

  • --output-dir=output: Defines the destination directory for storing the results of the SQL injection tests and any extracted data.

  • --batch: Enables non-interactive execution. All prompts are automatically answered using default options, which is ideal for scripting or automation.

3.4.3 PhpMyAdmin (port 80) - Remote Code Execution via Exposed Credentials

This vulnerability is classified as an authentication flaw, where valid credentials for the PhpMyAdmin web interface were disclosed in the system’s internal documentation (wiki). Specifically, the username and password were found to be root:sploitme, allowing an attacker to log in without the need for brute-force or guessing.


Figure 14: Setting options for the attack

To exploit this, the Metasploit module was configured with the appropriate options: the target port was set to 80, and the TARGETURI was specified as /phpmyadmin/. Upon providing the leaked credentials, the exploit successfully authenticated and attempted to deliver a Meterpreter payload.

This setup demonstrates how exposed or hardcoded credentials—even when documented internally—can lead to full remote code execution if combined with a vulnerable application like PhpMyAdmin.


Figure 15: Attack done. Meterpreter shell opened

3.4.4 Unreal IRC Service (port 6607) - Remote Code Execution via Backdoored Version

This vulnerability stems from the inclusion of a backdoor in a maliciously modified version of the UnrealIRCd source package (Unreal3.2.8.1.tar.gz). In November 2009, attackers replaced the original file on the official mirror with a tampered version that introduced a backdoor, allowing remote command execution with the privileges of the user running the IRC daemon.


Figure 16: Options for the Unreal IRC Service attack

To exploit this, the Metasploit module unix/irc/unreal_ircd_3281_backdoor was configured with the appropriate options: the RHOST was set to the target server, the payload selected was cmd/unix/reverse_perl, the target port was defined as 6697, and the LHOST was set to the attacker’s IP address (10.0.3.7). Once launched, the exploit sent a crafted backdoor command to the service.

This scenario demonstrates how the use of compromised software—especially from unverified or tampered sources—can introduce severe vulnerabilities, enabling unauthenticated attackers to gain remote code execution on affected systems.


Figure 17: Command execution shell

4 Conclusion

Collaborating on the Metasploit Framework over the past year has been a highly rewarding experience that provided me with a deeper understanding of how an open-source project operates from the inside, as well as the tools and workflows needed to participate effectively. In particular, I became proficient with Git and GitHub—skills that have become essential for any professional in the cybersecurity and software development fields.

Through my contributions, I integrated into an active and diverse open-source community, learned to follow its guidelines, and interacted with contributors and maintainers from different backgrounds. This experience allowed me to master the technical and collaborative processes necessary to contribute meaningfully, from creating well-structured documentation to reviewing code and suggesting improvements.

Working with Metasploit also helped me reflect on its place in the cybersecurity landscape. As mentioned by s4vitarr in a video I encountered [?], the true challenge and beauty of cybersecurity lies in creating your own modules rather than relying solely on those built by others. While Metasploit makes exploitation more accessible—including to people who perhaps should not have such easy access—it also serves as a powerful educational tool for those willing to go deeper.

Beyond technical skills, this project sparked my curiosity about the wider open-source ecosystem. I have since explored other projects, discovering valuable resources such as The Odin Project [?], a free platform for learning web development, which has enriched my skill set as a professional.

In summary, this internship not only allowed me to meet the stated objectives—integrating into an open-source community, mastering contribution tools, and understanding the implications of open-source choices—but also encouraged me to explore, learn, and critically reflect on my work. The year-long collaboration has been an important step in my growth as a cybersecurity practitioner and open-source contributor.
